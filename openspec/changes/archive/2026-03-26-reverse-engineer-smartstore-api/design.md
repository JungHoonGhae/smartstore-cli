## Context

`storectl`은 네이버 스마트스토어 판매자센터의 비공식 CLI 도구다. 현재 `internal/client/` 의 API 클라이언트는 `/api/products`, `/api/orders` 등 추정 기반 플레이스홀더 엔드포인트를 사용하며 실제 판매자센터 API와 통신하지 못한다.

네이버 스마트스토어 판매자센터(`sell.smartstore.naver.com`)는 SPA 프론트엔드로 구성되어 있고, 내부적으로 REST/GraphQL API를 호출한다. 이 API는 공식 문서화되지 않았으므로 브라우저 트래픽 캡처를 통해 리버스 엔지니어링해야 한다.

**현재 상태:**
- Playwright 기반 로그인 → NID_AUT/NID_SES 쿠키 획득은 이미 동작
- API 호출 시 세션 쿠키 적용 로직은 구현되어 있음
- 실제 엔드포인트와 요청/응답 포맷만 교체하면 동작 가능한 구조

## Goals / Non-Goals

**Goals:**
- 판매자센터 웹 프론트엔드가 사용하는 실제 API 엔드포인트 식별 및 문서화
- 각 도메인(상품, 주문, 통계, 정산, 문의, 리뷰)별 API 프로토콜 사양 작성
- 리버스 엔지니어링 결과 기반으로 Go 클라이언트 재구현
- 트래픽 캡처 재현 가이드 작성 (향후 API 변경 시 유지보수용)

**Non-Goals:**
- 네이버 커머스 API (공식 OAuth 기반 API) 지원 — 별도 change로 분리
- 상품 등록/수정 등 쓰기 API — 읽기 전용 원칙 유지
- 자동화된 API 변경 감지 시스템 구축
- 네이버 로그인 2FA/CAPTCHA 우회 메커니즘

## Decisions

### 1. 트래픽 캡처 방법: 브라우저 DevTools + HAR Export

**선택**: 브라우저 DevTools Network 탭에서 HAR 파일로 내보내기
**대안**: mitmproxy 투명 프록시
**이유**: mitmproxy는 TLS 인터셉션을 위한 CA 인증서 설치가 필요하고 설정이 복잡하다. 브라우저 DevTools는 추가 설치 없이 즉시 사용 가능하며, HAR 형식은 표준화되어 파싱이 쉽다. 캡처 가이드의 접근성을 높이기 위해 DevTools를 1차 방법으로, mitmproxy를 고급 대안으로 문서화한다.

### 2. API 사양 문서 형식: 엔드포인트별 Markdown + JSON Schema

**선택**: `docs/api/` 디렉토리에 도메인별 Markdown 파일, 요청/응답은 JSON Schema로 표현
**대안**: OpenAPI/Swagger YAML
**이유**: 리버스 엔지니어링된 비공식 API에 OpenAPI를 적용하면 공식 API로 오해될 수 있다. Markdown + JSON Schema는 가독성이 높고, 변경 발견 시 빠르게 수정 가능하며, 프로젝트의 Markdown 중심 문서화 관행과 일치한다.

### 3. API 클라이언트 구조: 도메인별 파일 유지 + 공통 요청 빌더 추가

**선택**: 현재의 `internal/client/product.go`, `order.go` 등 도메인 분리 구조 유지. 공통 요청 패턴(헤더, 쿼리 빌더, 페이지네이션 핸들러)을 `client.go`에 추가.
**대안**: 단일 거대 클라이언트 파일
**이유**: 도메인별 분리는 이미 잘 되어 있다. 실제 API 엔드포인트 교체 시 각 파일만 수정하면 되므로 변경 범위가 명확하다.

### 4. 응답 파싱 전략: 필요한 필드만 추출

**선택**: 실제 API 응답에서 CLI 표시에 필요한 필드만 Go struct에 매핑, 나머지는 무시
**대안**: 전체 응답 구조를 1:1 매핑
**이유**: 네이버 API 응답은 내부 프론트엔드용으로 불필요한 필드가 많을 수 있다. 필요한 필드만 매핑하면 API 응답 구조의 사소한 변경에 영향을 덜 받는다. `json.Decoder`의 기본 동작이 알 수 없는 필드를 무시하므로 자연스럽게 호환된다.

### 5. 세션 검증: 실제 엔드포인트 기반

**선택**: 현재 `/api/seller-profile` 대신 실제 판매자 프로필 API 엔드포인트로 교체
**대안**: 아무 엔드포인트나 호출해서 401 체크
**이유**: 프로필 엔드포인트는 보통 가볍고 부작용이 없으며, 판매자 ID 등 유용한 정보를 반환할 수 있다.

## Risks / Trade-offs

- **[API 변경 리스크]** 네이버가 프론트엔드 리팩터링 시 API 엔드포인트가 변경될 수 있음 → 트래픽 캡처 가이드를 문서화하여 누구나 재분석 가능하게 함. 버전별 변경 이력을 `docs/api/CHANGELOG.md`에 기록.

- **[인증 강화 리스크]** 네이버가 API에 추가 인증 메커니즘(CSRF 토큰, API 키, 요청 서명)을 도입할 수 있음 → 캡처 시 요청 헤더 전체를 분석하여 인증 관련 헤더를 모두 식별. 세션 저장 시 필요한 토큰도 함께 보관.

- **[법적 리스크]** 비공식 API 사용이 네이버 이용약관에 저촉될 수 있음 → README에 비공식 도구임을 명시. 읽기 전용으로 제한하여 플랫폼에 부하를 주지 않음. rate limiting 구현.

- **[불완전한 분석]** 특정 상태/조건에서만 나타나는 API 필드를 놓칠 수 있음 → 다양한 시나리오(주문 상태별, 상품 유형별)에서 캡처. `json:",omitempty"` 패턴으로 선택적 필드 처리.

## Open Questions

- 판매자센터가 GraphQL을 사용하는지 REST를 사용하는지 확인 필요 (캡처 후 결정)
- 페이지네이션 방식 (offset vs cursor) 확인 필요
- API 버전 헤더나 쿼리 파라미터 존재 여부 확인 필요
- Rate limiting 정책이 있는지 확인 필요
